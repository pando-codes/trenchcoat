- [ ] **Step 2: Create app/docs/[[...slug]]/page.tsx**

Renders any MDX page by slug, with TOC, title, description, and metadata.

```tsx
import { source } from "@/lib/source";
import {
  DocsPage,
  DocsBody,
  DocsDescription,
  DocsTitle,
} from "fumadocs-ui/page";
import { notFound } from "next/navigation";
import defaultMdxComponents from "fumadocs-ui/mdx";
import type { Metadata } from "next";

interface Props {
  params: Promise<{ slug?: string[] }>;
}

export default async function Page({ params }: Props) {
  const { slug } = await params;
  const page = source.getPage(slug);
  if (!page) notFound();

  const MDX = page.data.body;

  return (
    <DocsPage toc={page.data.toc} full={page.data.full}>
      <DocsTitle>{page.data.title}</DocsTitle>
      <DocsDescription>{page.data.description}</DocsDescription>
      <DocsBody>
        <MDX components={defaultMdxComponents} />
      </DocsBody>
    </DocsPage>
  );
}

export async function generateStaticParams() {
  return source.generateParams();
}

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params;
  const page = source.getPage(slug);
  if (!page) notFound();

  return {
    title: `${page.data.title} — Trenchcoat Docs`,
    description: page.data.description,
  };
}
```

Trenchcoat is a SaaS platform that gives engineering teams visibility into how they use AI agents. It ingests telemetry from Claude Code via a lightweight plugin, stores it in a managed database, and surfaces analytics through a dashboard — sessions, tool usage, token attribution, and cost.

## What Trenchcoat tracks

- **Sessions** — every time a developer opens Claude Code, a session is created
- **Tools** — every tool call (Read, Edit, Bash, etc.) is captured with name and duration
- **Agents** — subagent invocations are tracked separately so you can see which agents your team runs most
- **Tokens & cost** — input/output token counts per session, mapped to current model pricing

## How it works

```
Claude Code → Plugin hooks → Local JSONL → Batch push → Trenchcoat API → Dashboard
```

The plugin runs Python hooks on each Claude Code lifecycle event. Events are written locally first (under `~/.claude/telemetry/`) and flushed to the Trenchcoat API at the end of each session. No prompt content is ever captured — only metadata.

- [ ] **Step 1: Write plugin-sdk/overview.mdx**

```mdx
---
title: Overview
description: How the Trenchcoat Claude Code plugin works end-to-end.
---

The Trenchcoat plugin is a set of Python hook scripts that integrate with Claude Code's hook system. Each hook fires at a specific point in the Claude Code lifecycle and writes a telemetry event to a local JSONL file. At session end, all queued events are pushed to the Trenchcoat API in batches.

## Architecture

```
Claude Code lifecycle event
        │
        ▼
Hook script (Python, <3ms)
        │
        ├── Write event to ~/.claude/telemetry/events-YYYY-MM-DD.jsonl
        │
        └── Queue event for API push (if api_key configured)
                │
                ▼
        SessionEnd fires
                │
                ▼
        Batch POST to /api/v1/events  (fork, non-blocking)
```

## Privacy guarantees

The plugin is designed to be privacy-first:

- **No prompt content** is captured — `log_prompt_content` defaults to `false`
- **Tool inputs** are truncated to a 100-character preview by default
- **Tool results** are captured as size-in-bytes only — never the actual content
- All data is written locally first; you control what gets sent to the API via `api_key`

## Local storage

Events are stored under `~/.claude/telemetry/`:

| Path | Contents |
|---|---|
| `events-YYYY-MM-DD.jsonl` | Day-partitioned raw events (append-only) |
| `sessions.json` | Lightweight session index (started_at, status, duration) |
| `config.json` | Plugin configuration |
| `.push_queue.jsonl` | Events pending API push (cleared after successful flush) |
| `.pending/` | In-flight tool correlation state (pre/post matching) |

Events older than `retention_days` (default 30) are deleted on each `SessionEnd`.
```

The plugin is configured via `~/.claude/telemetry/config.json`. The file is created with defaults on first run. You can also run `/trenchcoat-connect` to set `api_url` and `api_key` interactively.

## Full config reference

```json
{
  "enabled": true,
  "privacy": {
    "log_prompt_content": false,
    "tool_input_preview_chars": 100,
    "log_tool_results": false
  },
  "api_url": "https://app.trenchcoat.com",
  "api_key": "ct_live_your_key_here",
  "push_batch_size": 100,
  "retention_days": 30,
  "webhook_url": null
}
```

## Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | boolean | `true` | Set to `false` to disable all telemetry collection |
| `privacy.log_prompt_content` | boolean | `false` | If `true`, captures full prompt text (not recommended) |

```mdx
---
title: Event Schema
description: All event types captured by the plugin, with their data payloads.
---

Every event sent to the Trenchcoat API has this envelope:

```json
{
  "ts": "2026-05-12T14:32:01.123Z",
  "event": "tool_use",
  "session_id": "abc123",
  "seq": 7,
  "data": { ... }
}
```

| Field | Type | Description |
|---|---|---|
| `ts` | ISO 8601 datetime with timezone offset | When the event occurred |
| `event` | string (enum) | Event type (see below) |
| `session_id` | string | Claude Code session identifier |
| `seq` | integer ≥ 0 | Monotonically increasing sequence number within the process |
| `data` | object | Event-specific payload (see per-type details below) |

## Event types

### `session_start`

Fires when Claude Code starts a session.

```json
{
  "event": "session_start",
  "data": {
    "cwd": "/Users/alice/projects/myapp"
  }
}
```

### `session_end`

Fires when Claude Code ends a session. `duration_ms` is computed from the session index.

```json
{
  "event": "session_end",
  "data": {
    "duration_ms": 42300
  }
}
```

### `tool_use`

Fires before a tool runs (maps from the internal `tool_start` event). The `input_preview` is truncated per `privacy.tool_input_preview_chars`.

```json
{
  "event": "tool_use",
  "data": {
    "tool_name": "Read",
    "correlation_id": "a1b2c3d4e5f6",
    "input_preview": "{\"file_path\": \"/src/index.ts\"}"
  }
}
```

### `tool_result`

Fires after a tool completes (maps from the internal `tool_end` event). Correlates with the matching `tool_use` via `correlation_id`. Only the size of the result is captured.

```json
{
  "event": "tool_result",
  "data": {
    "tool_name": "Read",
    "correlation_id": "a1b2c3d4e5f6",
    "duration_ms": 12.4,
    "result_size": 8192
  }
}
```

### `prompt_submit`

Fires when the user submits a prompt. Word count is computed locally; prompt text is never captured.

```json
{
  "event": "prompt_submit",
  "data": {
    "word_count": 24
  }
}
```

### `assistant_stop`

Fires when the main Claude process stops responding (end of a turn).

```json
{
  "event": "assistant_stop",
  "data": {
    "reason": "end_turn"
  }
}
```

### `subagent_stop`

Fires when a subagent finishes. Token and tool data are parsed from the subagent's transcript.

```json
{
  "event": "subagent_stop",
  "data": {
    "agent_type": "code-reviewer",
    "tool_counts": { "Read": 12, "Grep": 4 },
    "input_tokens": 18200,
    "output_tokens": 1400,
    "model": "claude-sonnet-4-6"
  }
}
```

### `pre_compact`

Fires before Claude Code compacts the conversation context. No data payload.

```json
{
  "event": "pre_compact",
  "data": {}
}
```

### `error`

Fires when the plugin encounters an error during hook execution.

```json
{
  "event": "error",
  "data": {
    "message": "Failed to read config: ..."
  }
}
```

## Accepted event types

The API accepts exactly these values for the `event` field:

```
session_start  session_end  tool_use  tool_result
prompt_submit  assistant_stop  subagent_stop  pre_compact
stop  error
```

Any other value is rejected with a `400 Bad Request`.

**Fires:** When Claude Code opens a new session.

**Script:** `hooks/session_start.py` — timeout 10s

**Reads:** `session_id`, `cwd`

**Writes events:** `session_start { cwd }`

**Side effects:** Initializes the session index entry in `sessions.json` with `started_at` and `status: "active"`.

---

## SessionEnd

**Fires:** When Claude Code closes a session cleanly.

**Script:** `hooks/session_end.py` — timeout 10s

**Reads:** `session_id`

**Writes events:** `session_end { duration_ms }` (duration computed from session index)

**Side effects:**
- Updates session index to `status: "ended"` with `ended_at` and `duration_ms`
- Flushes the push queue to the API (non-blocking fork) if `api_key` is configured
- Runs retention cleanup to delete JSONL files older than `retention_days`

**Fires:** Before every tool call (Read, Edit, Bash, etc.).

**Script:** `hooks/pre_tool_use.py` — timeout 5s

**Reads:** `session_id`, `tool_name`, `tool_input`

**Writes events:** `tool_use { tool_name, correlation_id, input_preview }`

**Side effects:** Pushes a correlation entry (tool name + timestamp) to `.pending/<session_id>.json` for matching with PostToolUse.

---

**Fires:** After every tool call completes.

**Script:** `hooks/post_tool_use.py` — timeout 5s

**Reads:** `session_id`, `tool_name`, `tool_result`

**Writes events:** `tool_result { tool_name, correlation_id, duration_ms, result_size }`

**Side effects:** Pops the matching correlation entry from `.pending/<session_id>.json` to compute duration.

**Fires:** When the user submits a prompt.

**Script:** `hooks/user_prompt_submit.py` — timeout 5s

**Reads:** `session_id`, prompt text (word count computed; text is never stored)

**Writes events:** `prompt_submit { word_count }`

---

## Stop (assistant stop)

**Fires:** When the main Claude process finishes a response turn.

**Script:** `hooks/stop.py` — timeout 5s

**Reads:** `session_id`, stop reason

**Writes events:** `assistant_stop { reason }`

**Fires:** When a dispatched subagent finishes.

**Fires:** Before Claude Code compacts the conversation context.

**Script:** `hooks/pre_compact.py` — timeout 5s

**Reads:** `session_id`

**Writes events:** `pre_compact {}`

Trenchcoat is designed for engineering teams. Each workspace has a shared view of sessions and analytics across all members.

## Inviting members

1. Go to **Settings → Team**
2. Click **Invite Member**
3. Enter the team member's email address
4. Select their role (see below)
5. Click **Send Invite**

The invitee receives an email with a link to accept the invitation. Pending invites are shown in the team list until accepted or expired.

## Roles

| Role | Permissions |
|---|---|
| **Member** | View sessions, analytics, and tool usage. Cannot manage API keys or invite others. |
| **Admin** | Full access — can manage API keys, invite members, remove members, and view all settings. |

New members default to the **Member** role. You can change a member's role at any time.

## Viewing member sessions

All sessions from all team members appear in the shared Sessions view. Sessions are identified by user account — use the **Directory** column to infer who ran a session if session attribution is unclear.

## Removing a member

Click **Remove** next to a member in the Team settings page. Their account is disconnected from your workspace immediately. Their historical sessions remain in your dashboard.

## Seat limits by plan

| Plan | Seats |
|---|---|
| Free | 3 members |
| Pro | Unlimited |

If you need to add more members on the Free plan, upgrade to Pro from **Settings → Billing**.
```
