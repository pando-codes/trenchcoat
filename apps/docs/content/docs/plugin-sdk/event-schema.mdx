---
title: Event Schema
description: Reference for the event types the Trenchcoat plugin sends and the data fields each event carries.
---

Every event the plugin sends shares a common base structure. Event-specific fields live in the `data` object, which varies by `event_type`. This page documents the full schema for all known event types.

## Base event structure

All events conform to this envelope:

```json
{
  "event_type": "tool_use",
  "session_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "timestamp": "2026-05-12T14:23:01.456Z",
  "data": { }
}
```

| Field | Type | Description |
|---|---|---|
| `event_type` | string | Identifies the type of event. See the table below for all valid values. |
| `session_id` | string (UUID v4) | Stable identifier for the current Claude Code session. All events within a session share the same `session_id`. |
| `timestamp` | string (ISO 8601 UTC) | When the event occurred, in UTC with millisecond precision. |
| `data` | object | Event-specific payload. Schema varies by `event_type`. |

The plugin sends events in batches via `POST /api/v1/events`. Each request body is an array of event objects: `{ "events": [ ... ] }`.

## Event types

### `session_start`

Fires when a new Claude Code session begins. Always the first event in a session.

```json
{
  "event_type": "session_start",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T14:23:00.000Z",
  "data": {
    "model": "claude-sonnet-4-6",
    "cwd": "/Users/alice/projects/my-app",
    "git_branch": "feature/dashboard"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `model` | string | The Claude model active at session start. |
| `cwd` | string | Working directory where Claude Code was launched. |
| `git_branch` | string \| null | Current git branch, if the working directory is a git repository. |

### `session_end`

Fires when the Claude Code session terminates. Always the last event in a session.

```json
{
  "event_type": "session_end",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T15:47:22.100Z",
  "data": {
    "duration_ms": 5062100,
    "total_tokens": 84320,
    "cost_usd": 0.253
  }
}
```

| Field | Type | Description |
|---|---|---|
| `duration_ms` | integer | Total session duration in milliseconds. |
| `total_tokens` | integer | Cumulative tokens used across all turns in the session. |
| `cost_usd` | number | Estimated session cost in USD, derived from model pricing. |

### `tool_use`

Fires immediately before a tool executes (via the `PreToolUse` hook).

```json
{
  "event_type": "tool_use",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T14:25:10.220Z",
  "data": {
    "tool_name": "Bash",
    "correlation_id": "9f86d081b8ac",
    "input_preview": "ls -la /Users/alice/projects"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `tool_name` | string | The name of the tool being invoked (e.g., `Bash`, `Edit`, `Read`). |
| `correlation_id` | string | Opaque ID that ties this `tool_use` event to its matching `tool_result` event. |
| `input_preview` | string \| null | A truncated preview of the tool input. Long inputs are trimmed to 100 characters by default (configurable via `privacy.tool_input_preview_chars` in `config.json`). |
| `spawner_id` | string | *Optional.* The `agent_id` (or skill activation ID) of the agent/skill this tool call happened under, if any. |
| `spawner_type` | string | *Optional.* Either `"agent"` or `"skill"`, paired with `spawner_id`. |
| `agent_id` | string | *Agent tool only.* Unique ID minted for this subagent spawn; carried onto the matching `tool_result` and the subagent's own `subagent_stop` event. |
| `edge_label` | string | *Agent tool only, optional, plugin ≥ 1.2.0.* One of `delegate`, `verify`, `critique`, parsed from a `[tc:...]` marker in the Task prompt. See [Edge labels](#edge-labels) below. |

### `tool_result`

Fires after a tool completes (via the `PostToolUse` hook).

```json
{
  "event_type": "tool_result",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T14:25:10.854Z",
  "data": {
    "tool_name": "Bash",
    "correlation_id": "9f86d081b8ac",
    "duration_ms": 634,
    "result_size": 1428,
    "is_error": false,
    "error_preview": null
  }
}
```

| Field | Type | Description |
|---|---|---|
| `tool_name` | string | The name of the tool that completed. |
| `correlation_id` | string \| null | Matches the `correlation_id` of the originating `tool_use` event, when one could be correlated. |
| `duration_ms` | number \| null | Wall-clock time from tool invocation to result, in milliseconds. |
| `result_size` | integer | Size in characters of the serialized tool result. |
| `is_error` | boolean \| null | Whether the tool returned an error result. `null` when the result shape doesn't affirmatively indicate success or failure. |
| `error_preview` | string \| null | Short (≤200 char) excerpt of the error content, populated only when `is_error` is `true`. |
| `spawner_id` | string | *Optional.* Same meaning as on `tool_use`. |
| `spawner_type` | string | *Optional.* Same meaning as on `tool_use`. |
| `agent_id` | string | *Agent tool only.* Same `agent_id` minted on the matching `tool_use` event. |
| `edge_label` | string | *Agent tool only, optional, plugin ≥ 1.2.0.* Same edge label recorded on the matching `tool_use` event, if any. |

### `subagent_stop`

Fires when a subagent spawned by the main Claude session completes.

```json
{
  "event_type": "subagent_stop",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T14:31:45.010Z",
  "data": {
    "agent_type": "general-purpose",
    "reason": "completed",
    "tool_counts": { "Read": 4, "Bash": 2 },
    "tool_count_total": 6,
    "turns": 5,
    "input_tokens": 8210,
    "output_tokens": 1340,
    "model": "claude-haiku-4-5",
    "agent_id": "3c9a1e7f2b04"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `agent_type` | string | The subagent type that ran (e.g., `general-purpose`), taken from the hook's `agent_type` field; defaults to `general-purpose` if absent. |
| `reason` | string | The stop reason reported by Claude Code for this subagent. |
| `tool_counts` | object | Map of tool name → invocation count, parsed from the subagent's transcript. |
| `tool_count_total` | integer | Total number of tool calls made by the subagent. |
| `turns` | integer | Number of assistant turns in the subagent's transcript. |
| `input_tokens` | integer | Cumulative input tokens across the subagent's turns. |
| `output_tokens` | integer | Cumulative output tokens across the subagent's turns. |
| `model` | string \| null | The model the subagent ran on, parsed from its transcript. |
| `agent_id` | string | *Optional.* The `agent_id` minted when this subagent was spawned (matches the `agent_id` on the spawning `tool_use`/`tool_result` pair), when it can be recovered from the pending stack. |

### `assistant_stop`

Fires each time the assistant finishes generating a response (via the `Stop` hook).

```json
{
  "event_type": "assistant_stop",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-05-12T14:25:11.001Z",
  "data": {
    "stop_reason": "end_turn",
    "tokens_used": 2340,
    "model": "claude-sonnet-4-6"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `stop_reason` | string | Why the assistant stopped. Common values: `end_turn`, `max_tokens`, `stop_sequence`. |
| `tokens_used` | integer | Tokens used in this assistant turn (input + output combined). |
| `model` | string | The model that generated this response. |

## Edge labels

*Requires plugin ≥ 1.2.0.*

When one agent spawns another via the `Agent` tool, the spawning agent can annotate the *reason* for the spawn by embedding a marker anywhere in the Task prompt:

```
[tc:delegate]
[tc:verify]
[tc:critique]
```

The plugin looks for a `[tc:<label>]` marker in the prompt passed to the `Agent` tool. If the label matches one of the three known values (`delegate`, `verify`, `critique`, case-insensitive), it is:

1. Recorded as the `edge_label` field on both the `tool_use` and the matching `tool_result` event for that `Agent` invocation.
2. Stripped from the prompt text before the prompt is truncated into `input_preview`, so the marker itself never appears in the stored preview and doesn't consume the truncation budget.

Markers that don't match one of the three known labels are left in place and ignored — no `edge_label` field is emitted. The convention only applies to the `Agent` tool; a `[tc:...]` marker in any other tool's input has no effect.

## Schema stability

The `data` field is stored as `jsonb` in Supabase and is not strictly validated at ingest time. This allows the schema to evolve without breaking older clients. Fields documented here are stable; additional fields may be added in future plugin versions without a major version bump. Removed or renamed fields will be announced in the changelog.
